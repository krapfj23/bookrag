BOOKRAG · ENGINEERING + DESIGN · INGESTION PLAN

EPUB → reading surface

What it takes to get the V3 reading experience — proper typography, drop caps, hyphenation, highlighted passages linked to chat cards — out of a real EPUB. The short answer: yes, largely. Below is the pipeline, what's realistic, and where the traps are.

An EPUB is a zip of XHTML + CSS + images + an OPF manifest. Everything on the page is already semantic HTML — we don't render it, we re-render it through our own typography layer. What we control is segmentation, normalization, and the chat-anchoring layer on top.

01 Pipeline

STAGE 1
Unzip & parse
Read .opf manifest. Order chapters by spine. Extract XHTML, CSS, images.
→
STAGE 2
Normalize HTML
Strip publisher CSS. Keep structure: p, h1–h6, blockquote, em, img, footnotes.
→
STAGE 3
Segment
Paragraphs → sentences → token spans with stable IDs (p12.s3). This is the anchor system.
→
STAGE 4
Paginate virtually
Run CSS multi-column layout at the design size. Snapshot which spans live on which page.
→
STAGE 5
Index for RAG
Embed paragraphs with their IDs attached. At query time, retrieved chunks cite back to p12.s3.
→
STAGE 6
Render + anchor
Draw pages through our typography. Chat cards attach to span IDs via DOM refs.

02 What's realistic — and what needs work

✓ Typography — font family, size, line height, justify, hyphenation. Our CSS, not the publisher's.
✓ Drop caps — applied via ::first-letter on chapter-opening paragraphs.
✓ Highlighted phrase anchoring — every sentence has a stable span ID.
✓ Chapter segmentation — spine gives us chapters directly.
✓ Images inline — normalize paths, lazy-load, preserve caption.
✓ Two-page spread layout — CSS columns with column-count: 2 and column-fill: auto.
! Page numbers — EPUBs don't have them. We generate them from virtual pagination at our chosen font size.
! Footnotes — typically epub:type="noteref". Need to render as popovers, not jump to back matter.
! Tables / equations — most novels lack them, but technical books do. Render, don't paginate through.
! DRM-locked EPUBs — out of scope. User supplies the file; we assume it's readable.

03 Gotchas

Publisher CSS is a minefield
Many EPUBs ship styles that fight our typography. Strip everything except structural elements and their roles — ignore font-family, colors, sizing from the source.
Stable IDs across re-flows
A chat card anchors to p12.s3. If we re-paginate (different font size), that span moves to a different page. The card follows the span, not the page.
Virtual "pages" aren't real
"Page 2 of 176" depends on font size, column count, viewport. Show location-like progress plus page-at-current-settings.
Footnotes interrupt flow
EPUB footnotes usually link to back-matter. For in-flow reading, intercept the link and render as a tethered popover next to the reference.
Image-heavy chapters
Some chapters are mostly plates. Don't try to paginate around them — let them take full spreads, and skip them for RAG indexing.
Fog-of-war + pagination
The cutoff word needs to stay on a page boundary. Virtual pagination must know where the reader left off, so the next spread opens at the right sentence.

04 Anchoring chat to text

Every sentence gets a stable ID at ingestion time. The chat panel stores cards keyed by those IDs. At render, we find the DOM node by [data-sid="p12.s3"] and draw the connector from the card to it. If the ID isn't on the current spread (reader scrolled or turned pages), the card shows ↑ scroll up or ← p. 1 and offers a jump.

At ingestion
<p id="p12">…
<span data-sid="p12.s3">Scrooge signed it.</span>…
In the index
Chunk embedding carries span IDs. Retrieval returns { quote, span: "p12.s3", chapter: 1 }.
On the page
Card pins to document.querySelector('[data-sid="p12.s3"]'). Scroll/page turn updates connector.

05 What to change in current implementation

REPLACE
Infinite scroll → paginated spread

Use CSS multi-column with column-count: 2, column-fill: auto, fixed height. Each "page" is one view of the two columns. Advance by translating the column track, not scrolling.

ADD
Span-level IDs at ingestion

Tokenizer walks the XHTML, assigns p{n}.s{m} to each sentence. Index chunks carry these IDs so every citation is anchorable.

ADD
Virtual pagination engine

Given a font size + viewport, compute which spans live on which page. Fast — runs on chapter load, caches per (font size, width).

ADD
Card-to-span positioning

On render, each chat card asks the DOM for its anchor span's bounding box and draws the connector. ResizeObserver keeps it in sync.

KEEP
Fog-of-war cutoff logic

The "already read up to span X" state is agnostic to the layout — works whether scroll or paginated. Just query: "is this span ≤ the reading cursor?"

KEEP
RAG retrieval layer

No change to embeddings or retrieval. We're only changing how retrieved chunks display — they now ship with span IDs.
