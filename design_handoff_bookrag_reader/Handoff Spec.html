BOOKRAG · v1 HANDOFF · FRONTEND SPEC

Reader handoff spec

One page. Decisions only. Points at the designs, ingestion plan, and design system for visual and data specifics. Read this, then build; refer back for what, not how.

Scope: Build the reader UI in the browser. The ingestion pipeline and storage layer already exist in the backend code — see CLAUDE.md for pipeline, kuzudb, and embedding details. This spec is for the reading + chat surface only.

01 Product decisions

Reading + chat

Chat direction
V3 Inline — margin note-cards.locked
See EPUB Visualizer v6.html → "Chat — four directions" section, artboard CV3.
Reading mode
Ambitious — ambient paper, pacing label, note-peek on hover.locked
See visualizer → "Reading mode · sidebar hidden" → R2on and R2hover.
Overflow strategy
O2 Collapse only — older cards shrink to 1-line summaries; latest expanded.locked
Visualizer → "V3 Inline · overflow strategies" → O2.
Platform
Browser only (desktop-first). No mobile in v1.locked
Reading mode persistence
Per-book in localStorage. Key: bookrag.reading-mode.{bookId}.
Keyboard
Arrow keys only for page turns. No space, no J/K in v1.

Behaviors

Duplicate ask
Ask on a phrase that already has a card → scroll/focus to that card, open its follow-up composer. Don't make a new card.
Notes vs. asks
Separate entities. A note has a body only (user-authored prose). An ask has a question + assistant answer + optional follow-ups.
Edit / delete
Only from a card detail view. Not inline on the margin card. The margin card is read-only surface.
Card detail view not yet designed. Flag for a follow-up pass.
Cross-page cards
Yes, but collapsed by default. Shown as the 1-line summary form (same as O2 for older cards on the same spread).
Visualizer → atomic states → S7 shows the cross-page tag. Combine with O2 collapse styling.
Fog of war
Ask-ability is gated by the reading cursor. Anything before the cursor is fair game. Anything after is not.
The reading cursor is stored in localStorage and advanced when the user page-turns past a sentence span.
Streaming
Token-by-token (ChatGPT style). The S3 atomic state already mocks the typing cursor and "thinking" placeholder.
Touch / hover
Desktop only in v1. Hover-to-peek for notes is fine. No touch fallback needed yet.

02 Data model (frontend-facing)

The frontend consumes the backend's EPUB corpus (chapters + sentence spans) and reads/writes cards against sentence anchors. Schema is suggestive — confirm exact field names with the backend layer.

// Anchor format from ingestion: "p{paragraphIndex}.s{sentenceIndex}" // e.g. "p12.s3" = paragraph 12, sentence 3 (0- or 1-indexed — confirm) type Card = { id: string; // uuid bookId: string; anchor: string; // "p12.s3" quote: string; // exact text of the anchored span chapter: number; // from EPUB spine kind: "ask" | "note"; createdAt: ISO8601; updatedAt: ISO8601; } & ({ kind: "ask"; question: string; answer: string; // streaming target followups: Array<{ question, answer }>; } | { kind: "note"; body: string; }); type ReadingCursor = { bookId: string; anchor: string; // last sentence the user has "read past" };

03 Backend dependencies (not in this spec)

These are owned by the backend code / CLAUDE.md. Frontend assumes them:

  • EPUB ingestion pipeline — already implemented. Produces chapters, paragraphs, and sentence-level anchors. owned: backend
  • Sentence ID generation — "who generates p{n}.s{m}" is a backend question. Frontend receives them pre-assigned. owned: backend
  • Storage + embeddings — kuzudb per CLAUDE.md. Frontend talks to the backend via whatever API that layer exposes. owned: backend
  • RAG retrieval — given a question + book scope, return chunks with their anchor IDs. Frontend renders the citation. owned: backend
  • Scope — single user, single device, local-only. No sync, no sharing in v1.
  • DRM — unencrypted EPUBs only. User provides the file; no DRM removal in v1.

04 URL + state

URL shape
/book/{bookId} — book ID only. Everything else in localStorage.
localStorage keys
bookrag.cursor.{bookId} — reading cursor anchor
bookrag.reading-mode.{bookId} — boolean
bookrag.font-size.{bookId} — if we expose font-size controls later
bookrag.last-book — for library landing
Cards
Persisted by backend (kuzudb). Not in localStorage.

05 Build order

Suggested sequence. Each step produces a working artifact you can demo.

  • 1. Reading surface — paginated two-page spread with real EPUB content. Virtual pagination by font size. Arrow-key turns. No chat yet. ~2 days
  • 2. Sentence anchors rendered — every span gets data-sid="p12.s3". Verify backend IDs line up. ~0.5 day
  • 3. Fog-of-war cursor — track reading cursor; dim/blur text after it. Advance on page-turn. ~1 day
  • 4. V3 Inline margin column — render cards keyed to visible anchors. Empty state + invitation. ~2 days
  • 5. Selection → Ask flow — highlight phrase, toolbar, create card. Token-stream the answer. ~2 days
  • 6. Notes — same flow, no LLM, body-only card. ~1 day
  • 7. Card states — streaming, long-answer scroll, thread follow-ups, off-screen, cross-page. ~2 days
  • 8. O2 overflow — collapse older cards to 1-liners, expand latest. ~1 day
  • 9. Reading mode (ambitious) — toggle, ambient paper, note-peek on hover, sidebar collapse. ~1.5 days
  • 10. Card detail view — edit + delete. Design pass needed first. ~1 day after design

06 Known gaps / future passes

Card detail view not designed yet
Edit + delete only happen there, per decision. Before build step 10, I should do a quick design pass on this screen.
Mobile / touch
Out of scope in v1. Ambitious reading mode's hover-peek has no touch fallback yet — revisit when mobile is on the roadmap.
Virtual pagination engine
Not yet prototyped. The ingestion plan describes it; it's real engineering work, not a design task. Build step 1 is where this bites.
Card-to-span connector rendering
The margin cards need to draw a subtle line/arc to their anchor span. The mocks use static SVGs; production needs live DOM measurement + ResizeObserver. Flag during step 4.
