# Slice 2 Early Review — upload-pipeline-status

**Date:** 2026-04-21
**Stage:** early (pre-Generator)
**Reviewer:** Evaluator agent (superpowers:code-reviewer)

## Summary

PROCEED-WITH-FIXES. The plan is unusually thorough (every step has exact code + expected command output + a scoped commit) and the test layer covers 9 of 10 ACs with reasonable rigor. Three real problems should be patched before the Generator is dispatched: the polling-cadence assertion is loose enough to pass a broken implementation, the LibraryScreen refetch test can't distinguish a `[]` dependency list from `[pathname]`, and the `UploadScreen.test.tsx` "maps the error" flow depends on error text that is only enforced indirectly by the `api.test.ts` suite.

## AC <-> test coverage

- **AC 1 (Upload/Library routing without reload; Reading inert).** Covered by `NavBar.test.tsx` (`"marks Upload active when the route is /upload"`, `"Library and Upload are real links with href attributes"`, `"Reading stays inert..."`) and `App.test.tsx` (`"renders LibraryScreen at /"`, `"renders UploadScreen placeholder at /upload"`). E2E `upload.spec.ts` (`"navigates from Library -> Upload -> Library without reloading"`) additionally asserts `page.toHaveURL(/\/upload$/)`. Good.
- **AC 2 (Upload screen header + idle Dropzone + empty pipeline area).** Covered by `UploadScreen.test.tsx` (`"renders the header, tagline, and idle Dropzone"`) and E2E `"Upload screen renders the idle Dropzone"`. Note: no test explicitly asserts that the pipeline panel is NOT rendered in the idle state. A buggy implementation that always renders the panel with empty rows would still pass. CONCERNS.
- **AC 3 (drag/drop OR file-input POSTs multipart to /books/upload and transitions to uploading).** Covered by `Dropzone.test.tsx` (`"calls onFile when a file is dropped"`, `"calls onFile when a file is selected via the hidden input"`), `api.test.ts` (`"POSTs multipart/form-data to /books/upload..."` asserts `FormData` with `file` field), and `UploadScreen.test.tsx` (`"uploads the dropped file, then polls and renders the 7 stages"`). The "transitions to uploading" visual state is NOT directly asserted in `UploadScreen.test.tsx` (no assertion on `data-state="uploading"` or "Uploading..."); it's only implicitly exercised by the filename rendering. CONCERNS (minor).
- **AC 4 (success -> Dropzone done + pipeline panel with book_id subtitle + 7 labelled rows in order).** Covered by `UploadScreen.test.tsx` (`"uploads the dropped file, then polls and renders the 7 stages"`) — but row order is not asserted (the test queries each label in isolation; a reversed-order bug would still pass). Dropzone `done` state with filename is asserted via `screen.getAllByText("a-christmas-carol.epub")`, but not via `data-state="done"` on the dropzone root. E2E spec `"a successful upload renders the 7 pipeline stages"` also covers this but likewise does not assert order. CONCERNS (minor).
- **AC 5 (2000 ms polling cadence, each poll updates every row's badge).** Only partially covered. `UploadScreen.test.tsx` advances time by 2000 ms once and asserts `statusSpy` has been called, but never asserts "exactly 1 call after 2000 ms" or "3 calls after 6000 ms" — a naive implementation that polls every 100 ms would also pass. The "each poll updates badges" claim is effectively tested because the first mocked response has `parse_epub: complete` and the test asserts the rows render, but it does not assert the badge STATE changes over two polls with different payloads. CONCERNS.
- **AC 6 (polling halts on ready_for_query OR first failed stage).** Covered by `UploadScreen.test.tsx` (`"stops polling and renders 'Back to Library'..."` snapshots `statusSpy.mock.calls.length` before advancing 4000 ms and asserts it stays equal; `"stops polling and shows an error banner..."` does the same). Good.
- **AC 7 ("Back to Library" navigates to / and Library re-fetches GET /books).** Navigation is covered by `UploadScreen.test.tsx` (`"...renders 'Back to Library'..."` asserts `href="/"`). Library refetch is covered by the appended test in `LibraryScreen.test.tsx` (`"re-fetches /books when the route is re-entered..."`). HOWEVER — see Risk #2 below: this test uses two separate `<MemoryRouter>` mounts, which forces a REMOUNT of `LibraryScreen`. It would pass with either `useEffect(..., [])` or `useEffect(..., [pathname])`, so it does not actually verify the intended dependency key. FAIL for strict coverage of "re-fetches on route entry"; PASS for "fetches on mount".
- **AC 8 (failed stage -> error row with sanitized message, inline banner, polling halts, other stages keep last-known state).** Partially covered. `UploadScreen.test.tsx` (`"stops polling and shows an error banner..."`) asserts `role="alert"` contains "pipeline failed" AND "OOM killed" is visible, plus polling halts. `PipelineRow.test.tsx` (`"shows meta when state is 'error'"`) asserts the row shows the meta string and has `aria-label="failed"`. The "other stages retain their last-known state" phrase is NOT asserted anywhere — if the implementation blanks the other rows on failure, no test catches it. CONCERNS.
- **AC 9 (400 -> exact copy "Only .epub files are accepted"; 413 -> exact "File too large (max 500 MB)"; 429 -> exact "Too many pipelines running, try again later"; other -> backend detail; Dropzone returns to idle).** Exact-string coverage in `api.test.ts` (`"maps 400..."`, `"maps 413..."`, `"maps 429..."`, `"falls back to backend detail text..."`, `"uses a generic message when JSON parsing fails"`). Good. But "Dropzone returns to idle" is NOT asserted — in fact the plan's implementation leaves the Dropzone in `state="error"`, NOT `idle`. The `UploadScreen.test.tsx` error test asserts `role="alert"` contains the message; it never asserts `state="idle"`. This is a PRD<->plan conflict that the tests paper over. FAIL for AC 9's "Dropzone returns to idle" clause (treat as a PRD ambiguity worth resolving pre-Generator).
- **AC 10 (npm test and pytest -v both pass, no regressions, no new backend endpoints).** Structurally covered by Task 9 Steps 9.1 and 9.4 in the plan. No pre-Generator test artifact can verify this.

## Plan quality

Overall the plan is high-signal. Every task has exact file paths, exact code (full file replacements where reasonable), and exact commands with expected output. Minor issues:

- **Task 1 Step 1.7** instructs replacing "the single line `<NavBar active="library" />` in `LibraryScreen.tsx`" — but at slice 1 close, the NavBar call already reads `<NavBar />` (no `active` prop in the committed slice-1 `LibraryScreen.tsx`). The Generator should not trip on this, but the plan's premise is slightly stale. Confirm the Generator does not mechanically search-and-replace a string that is not present.
- **Task 2 Step 2.3** — the `uploadBook` implementation calls `resp.json()` once even when `resp.ok` is false, and the body is consumed twice if the first read was partial. Node `fetch` Response bodies are single-shot; the test `"uses a generic message when JSON parsing fails"` mocks `json: () => Promise.reject(...)` which is compatible with the `try/catch` path, so this is safe in practice. No fix needed, just flagging.
- **Task 6 Step 6.3** — the effect dependency arrays use `[phase.kind === "tracking" ? phase.book_id : null]` which the React ESLint `exhaustive-deps` rule will flag. The plan includes `// eslint-disable-line` in the actual source (see `UploadScreen.tsx` at HEAD) but the plan's embedded code block does NOT include those directives. The Generator may run into a lint warning if a lint gate is added later.
- **Task 6 Step 6.1** — the test `"shows the mapped error in the Dropzone when uploadBook rejects"` asserts `screen.getByRole("alert")` contains the error. The Dropzone renders the error string inside a `<div role="alert">` — but the plan's `UploadScreen` implementation has a SECOND `role="alert"` for the failed-pipeline banner. When both render simultaneously (e.g. if a later regression keeps the error Dropzone state AND a failed stage) `getByRole("alert")` would throw. Not a risk for the test-as-written, but a design fragility.
- **Task 7 Step 7.1** embeds `npx playwright install chromium` inline. Playwright browser downloads are large and may hit network quotas in sandboxed CI; note as a PROCEED-WITH-FIXES candidate only if the Generator is running offline. In this environment it should be fine.
- **Task 8 Step 8.1** test note acknowledges: "the rerender with a different `<MemoryRouter>` forces React to remount, which is fine because production navigation ... also changes `location.pathname`." This is the author explicitly admitting the test is not strict. See Top Risk #2.
- **Task 9 Step 9.4** uses `/Users/jeffreykrapf/anaconda3/bin/pytest` (hard-coded absolute path). Fine for this machine; won't generalize. Acceptable for this single-user project.
- No task is hand-wavy. Every task has a named test gate and a commit step.

## Test meaningfulness

- **`Dropzone.test.tsx`** — PASS. Eight tests cover all five visual states, both input paths (drop + change), and the `busy` gate. Reasonable behavior coverage.
- **`StatusBadge.test.tsx`** — PASS. Six tests cover all five states, verify `aria-label` exact-matches the PRD vocabulary (including `"error" -> "failed"`), and verify custom label override and the pulsing dot data attribute.
- **`PipelineRow.test.tsx`** — PASS. Four tests cover meta rendering in done/error states, StatusBadge integration, and the idle no-meta case via `data-pipeline-meta`.
- **`UploadScreen.test.tsx`** — CONCERNS. Five tests, correct high-level shape, but (a) polling-cadence rigor is low — only the termination branches (`ready_for_query` / `failed`) are tightly asserted; the 2000 ms interval is only shown to call `fetchStatus` after 2000 ms, not on that cadence specifically; (b) no assertion on the Dropzone's visual state after upload error (AC 9's "returns to idle"); (c) no assertion on row ORDER (AC 4); (d) the book_id-visible-before-first-poll assertion is implicit because `mockResolvedValue` always returns a response — if the real implementation waited for the first poll before rendering the book_id subtitle, the test would still pass.
- **`NavBar.test.tsx`** — PASS. Explicitly asserts `<Link>` `href` attributes (`"/"` and `"/upload"`), `aria-current="page"`, and that Reading click is a no-op. Good.
- **`LibraryScreen.test.tsx`** — CONCERNS. The appended `"re-fetches /books when the route is re-entered..."` test uses two `<MemoryRouter>` mounts; this causes a remount, so the test passes with `useEffect(..., [])` — defeating the intent of the assertion. A test that wraps a single `<MemoryRouter>` around a parent that calls `useNavigate` to push onto the history stack would actually verify the dependency key. FAIL on intent; PASS on letter.
- **`App.test.tsx`** — PASS. Asserts the LibraryScreen and UploadScreen render at their respective routes under `MemoryRouter`. Two tests, adequate.
- **`api.test.ts`** — PASS. Strong. Each of the four PRD error-code mappings has an exact-string assertion. `FormData` and `File` identity are checked. The one subtle issue is that the `"uses a generic message when JSON parsing fails"` test uses a regex `/upload failed/i` which matches the fallback `Upload failed (${status})` — good.
- **`upload.spec.ts`** — PASS (hermetic). All four E2E tests use `page.route()` interceptors so they need no live backend. The Playwright config has a `webServer` block pointing at `npm run dev`, so the Vite dev server is auto-started and torn down. The 400-error E2E test verifies the exact PRD copy. Good. One minor concern: the "a successful upload renders the 7 pipeline stages" test relies on `mockStatusRunning` being set up before the hidden-input upload fires; if the Dropzone's click-to-browse logic happened to trigger a race, a missed mock would hang. Not a real risk given `page.route()` is installed before `page.goto()`.

## Top risks (ranked)

1. **Loose 2 s polling-cadence assertion in `UploadScreen.test.tsx` (AC 5).** A Generator that polls on a 500 ms or 1 s cadence, or that polls on every render rather than via `setInterval`, would still pass. Mitigation: tighten the successful-upload test to advance timers by 6000 ms and assert `statusSpy.mock.calls.length` equals either 3 or 4 (depending on whether an immediate poll is intended), and that no call happened between 0 ms and 1999 ms if the first tick is interval-gated.
2. **`LibraryScreen.test.tsx` refetch test can't distinguish `[]` from `[pathname]` in the effect deps.** The test remounts the component via two `<MemoryRouter>` instances, which always triggers a fresh mount and a fresh fetch. Mitigation: restructure the test to mount a single parent that contains both `<LibraryScreen />` and a `useNavigate()` caller, then invoke `navigate("/upload")` followed by `navigate("/")` and assert `fetchMock` was called at least twice. Alternatively add a second test that asserts the effect dep changes by wrapping in a `<Routes>` / `<Link>` harness.
3. **AC 9's "Dropzone returns to idle" on upload error is not asserted AND contradicts the plan's implementation** (plan puts Dropzone in `state="error"` which is semantically correct but textually violates the PRD). Mitigation: pick one — either update the PRD to say "Dropzone enters error state with the mapped message" (matching the plan + tests), or update the plan + tests to reset the Dropzone to idle after an inline banner renders the error. Decide before the Generator runs.
4. **No order assertion on the 7 pipeline rows (AC 4).** A Generator that alphabetizes or mis-orders the stages would still pass. Mitigation: add a single assertion along the lines of `expect(screen.getAllByRole(...).map(n => n.textContent)).toMatchInlineSnapshot(...)` or iterate `STAGE_DISPLAY` and assert each row's position.
5. **AC 8's "other stages retain their last-known state" clause has no test.** A Generator that blanks non-failed rows on failure would still pass. Mitigation: the fail-banner test already asserts `parse_epub: complete (0.4s)` in the mock; extend it with `expect(screen.getByText("0.4s")).toBeInTheDocument()` AFTER the failed banner renders, so a regression that blanks the meta would fail.

## Pre-generation fixes

1. **Tighten `UploadScreen.test.tsx` polling cadence.** In the `"uploads the dropped file, then polls and renders the 7 stages"` test, change the single `advanceTimersByTimeAsync(2000)` + `expect(statusSpy).toHaveBeenCalled()` pair to: advance by 6000 ms and assert `statusSpy.mock.calls.length >= 3 && <= 4`. This pins the cadence to ~2 s without requiring exact 3-or-4 (depending on whether an immediate tick is intended).
2. **Strengthen `LibraryScreen.test.tsx` refetch test.** Replace the two-`<MemoryRouter>` rerender with a harness that uses a single `<MemoryRouter>` + `<Routes>` + `<Link>` and have the test click a link to `/upload` then a link back to `/`; then assert `fetchMock` was called twice. That test would FAIL if the dep list is `[]`.
3. **Decide AC 9's "Dropzone returns to idle" semantics.** Either edit the PRD line to match the plan ("Dropzone enters `error` state with the mapped message"), or edit the plan's `UploadScreen` `Phase` error branch so `dropzoneState("error")` returns `"idle"` while still showing the banner. If the latter, update `Dropzone.test.tsx` to no longer require `state="error"` to render the alert — currently that path only runs for `state="error"`.
4. **Add a row-order assertion in `UploadScreen.test.tsx`.** In the success path: `const titles = screen.getAllByText(/(Parse EPUB|Run BookNLP|Resolve coref|Discover ontology|Review ontology|Cognee batches|Validate)/); expect(titles.map(n => n.textContent)).toEqual(["Parse EPUB", "Run BookNLP", "Resolve coref", "Discover ontology", "Review ontology", "Cognee batches", "Validate"]);`
5. **Add a "last-known state preserved on failure" assertion.** In the fail-banner test, after `await waitFor(() => expect(screen.getByRole("alert"))...)`, add `expect(screen.getByText("0.4s")).toBeInTheDocument();` so the passing `parse_epub` stage's duration survives the failure.

## Recommendation

Proceed directly to the Generator, but apply the five small test-level and spec-level adjustments above first. None require rewriting the plan — they are additive or clarifying changes to test files and one sentence in the PRD. The plan itself is well-scoped, every task has exact code and command gates, and the tests cover 9 of 10 ACs substantively. The two big latent failure modes (loose polling cadence, stale refetch test) are easy to tighten before dispatch, and doing so will save a REVISE loop after generation. If the orchestrator prefers to skip these fixes, the slice will almost certainly still ship, but AC 5's guarantees and the pathname-deps regression in AC 7 will be underspecified in the test suite.
