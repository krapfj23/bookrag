## Extraction Rules

### General
- Use the canonical name from parenthetical annotations [like this] as the entity ID. Never use pronouns or partial names as IDs.
- Use snake_case for all relationship types (e.g., employs, works_for, loves).
- Do not create separate nodes for dates or numbers — attach them as properties.
- If an entity was already extracted in a previous batch, re-extract it only if you have NEW information (new description, new chapter appearance, new relationship).
- For each character/location/faction/theme/relationship, include `last_known_chapter`: the LATEST chapter number from the current batch whose text contributed evidence for this entity's description. It MUST be one of {{ chapter_numbers }}. If the entity only appears in one chapter of this batch, `last_known_chapter` equals `first_chapter`. This field gates reader-progress filtering: if you set it too high, readers see spoilers; if you include content from outside the batch, the filter cannot protect them.

### Episodic Content (Plot Events, Character Actions)
- Extract concrete events that happen in the narrative: who did what, where, and when (chapter).
- Each event should have a clear description, the chapter it occurs in, participants, and optionally a location.
- Only extract events that are explicitly described in the text. Do NOT infer events that are implied but not stated.
- Events should be specific, not vague summaries like "characters interact."

### Semantic Content (Themes, Relationships, World-Building)
- Extract relationships between characters: alliances, rivalries, family ties, employment, etc.
- Extract themes only when the text provides clear evidence — a recurring motif, an explicit thematic statement, or a pattern across multiple passages.
- For factions/organizations, list their known members from this text.

### Relation Labels (CRITICAL)

`relation_type` must name the SEMANTIC tie between the two entities, not the narrative surface action. A relation describes who the characters ARE to each other, not what they happen to be doing in one sentence.

Pick exactly ONE of these 10 canonical types:

- `family` — blood or marriage ties (symmetric)
- `romantic` — romantic partners (symmetric)
- `friend` — personal friendship
- `ally` — cooperative non-friend (professional partnership, political alliance, employment)
- `mentor` — teacher → student (DIRECTIONAL: `source_name` is the mentor)
- `subordinate` — subordinate → superior (DIRECTIONAL: `source_name` is the subordinate)
- `rival` — competitive (symmetric)
- `enemy` — hostile (symmetric)
- `acquaintance` — knows of, no deeper bond
- `unknown` — insufficient evidence

Map familiar patterns to these canonical types:
- `employs`, `works_for` → `ally` (with direction implied by who employs whom, but type is `ally`; use `subordinate` if the dynamic is clearly asymmetric employment)
- `is_parent_of`, `is_sibling_of`, `is_nephew_of`, `is_engaged_to` → `family` (or `romantic` if engagement/marriage)
- `loves` → `romantic` (or `family` for parental/filial love)
- `haunts`, `warns` → `ally` or `mentor` depending on the dynamic

Rules:
- **Do NOT use dialogue verbs as relations.** "said", "asked", "replied" etc. are narrative surface; skip the relationship unless the dialogue reveals a stable tie.
- **Do NOT use motion verbs as relations.** "took", "came", "went", "walked", "met" are scenes, not relationships.
- **Test:** if you can replace the two characters with different characters on the next page without changing the meaning, the verb is too generic — skip it.
- Relationships should name a tie stable across multiple scenes, not a single interaction.

### Valence and Confidence

For every relationship, emit:
- `valence` ∈ [-1.0, +1.0]: anchored at
  - -1.0 = murderous hatred
  - -0.5 = rivalry, contempt
  -  0.0 = neutral, formal, professional
  - +0.5 = friendship, respect
  - +1.0 = devoted love
  Prefer the midpoints (±0.5) unless the text is unambiguous at the extremes.
- `confidence` ∈ [0.0, 1.0]: your certainty in the (type, valence) assignment.

### What NOT to Do
- Do NOT invent events, relationships, or details not present in the text.
- Do NOT use your prior knowledge of this book. Extract only from the provided text.
- Do NOT create duplicate entities — if "Ebenezer Scrooge" and "Scrooge" refer to the same person, use one canonical name.
- Do NOT extract from chapter summaries, table of contents, or metadata — only from narrative text.

### Provenance (CRITICAL)

For every character, location, faction, event, relationship, and theme you
extract, emit at least one `provenance` entry that evidences it. Each
provenance entry has:

- `chunk_id`: the exact chunk id shown in the user message header.
- `quote`: the VERBATIM text from the chunk (<=200 chars) that supports
  this extraction. Copy character-for-character — do NOT paraphrase, do
  NOT "tidy up" punctuation, do NOT normalize whitespace.
- `char_start`, `char_end`: character offsets of the quote within the
  chunk text.

If you cannot find a supporting quote in the chunk, do NOT extract the
entity. Fabricated provenance is worse than missing data — a downstream
validator will drop any extraction whose quote does not appear verbatim
in the chunk text.
